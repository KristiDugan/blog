Git on Board with Saving Keystrokes

How to create an alias to save time and increase efficiency

Amy McGowan

There’s always something new to learn about git. In this tutorial I want to share a great git feature with you. But this feature is not another crazy and complex command that you have to memorize. Instead, this is a feature that lets you memorize less. Intrigued? Read on.

There’s one thing about the daily use of git can get quite cumbersome: all the repetitive typing. Git status, git add -A, git commit -m “message”, git status, git checkout master, git checkout branch, git status, git add -A, git commit -m “message”... There’s gotta be a better way!

And there is.

Let’s give a warm welcome to Git Aliases. :clapping hands:

What is a Git Alias?

A Git Alias is as it sounds: a shortcut for a longer command. The basic use for setting up an alias is to shorten common commands that you type everytime you use git. For example, with a git alias you only need to type git co instead of git checkout or git st instead of git status. But the magic really happens when you start using aliases for more complex, multi-step commands.

(Note: There’s also another thing called bash aliases. That’s similar, but different).

This tutorial assumes you are already familiar with git and the command line. If you are and you’re ready to streamline your git flow, then let’s dive on in!

How to Create a Git Alias

If you’re already using git, then you have a configuration file called .gitconfig that lives in your $HOME directory. Out of the box, it has some basic user info such as your name and email address, but we’re going to add to that today. Let’s use git status as our example. To set an alias for git status, you simply type this command into your terminal git config --global alias.st status Here the --global flag sets the alias so that it will be available in any git repository on your machine. alias sets the alias. Here the co is whatever you want the shortcut to be, and the last word is what it is taking the place of. (if it the command that is being shortened is longer than one word, it needs to be in quotes). Now when you are in the terminal and want to check git status, you can just type git st. These are a few more basic ones that you might find useful git config –global alias.co checkout git config –global alias.cob ‘checkout -b’ git config –global alias.com ‘checkout master’ git config –global alias.ci commit

Let’s Step It Up

By far, one of the series of commands that I find most tedious is this: Git add -A Git commit -m “message” So let’s set an alias that combines both of these commands into one Git config –global alias.ac ‘!git add -A && git commit -m’ (The ! is necessary when stringing together multiple commands). Now when you want to commit all your files, including untracked files, and want to make a commit, you can type Git ac “message” That’s so much shorter, right?!

Bonus Aliases

Let’s look at two more helpful aliases.
Publish
Let’s imagine you’re making your first push on a new feature branch. Personally, I regularly get an error message like this after typing git push [ -set upstream error] And then I either have to use the mouse to copy/paste that line or I proceed to type out Git push -u origin It’s not super long...but it’s not short either. And if you’re working with many branches a day, the push-error-copy-paste-push workflow is not too efficient. Instead, you can create this alias. It sets the upstream remote branch with the same name as the locoal branch, tracks it, and pushes the current branch all in one quick step. Then when you’re ready to push on your new branch, all you have to type is git publish Git config –global alias.publish "!git push -u origin $(git !git rev-parse --abbrev-ref HEAD)" The first part of the replaced command looks the same as what appeared in our error message, but what about the last part in parentheses? Shouldn’t that just be the ? It is! This is a helpful combo of git commands that retrieves the name of the current branch. Now your workflow just changed from this ... Git push [error message] Copy/paste Success To ... Git publish
unstage
Another use case for git aliases is not only to simplify commands, but even to rename them. For example, git reset HEAD -- is used to remove a file from the staging area. It’s not necessarily a long command, but it’s not an intuitive name either. Since we are unstaging files, why not just call it unstage? Git confit –global alias.unstage “reset HEAD –” Now you can simply type Git unstage /file/path.html Instead of Git reset HEAD -- /file/path.html That’s an improvement in my book!

Some final tips

Thank you for taking the time to read this tutorial. I hope it has helped you to find a more efficient git workflow. You have learned about your .gitconfig file, how to set up basic git aliases, and have seen some examples of slightly more complex aliases that combine multiple commands into one alias. Before we go, I’ll leave you with a couple final alias tips.

First of all, only create an alias for what you commonly use and for what will actually be helpful. There are a number of git alias lists out there, but there’s not need to add them all. It just doesn’t make sense to memorize a lot of aliases, especially ones that aren’t used on a regular basis. Start with a few obvious basics, get used to using those, and then only create others as you realize they would be useful to have. Lastly, you have added your aliases one by one, but how do you review what you have? You have a few options. In the command line, you can enter
  • Open ~/.gitconfig – will open the file in your default text editor
  • Code ~/.gitconfig – will open the file in VS Code if that’s what you use and you’ve set up the code command
  • Git config --list --show-origin - will show the contents of your config file. This will also show the path to your config file if you’re not sure where it is.
